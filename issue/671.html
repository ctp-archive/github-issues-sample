Arizona `negativeIncrease` is incorrectly identified

Issue number 671

rgerkin opened this issue on July 24, 2020 at 4:27 pm

Labels Data quality

Open in Github

State or US: Arizona

Describe the problem Your API is currently using data like this and reporting positiveIncrease to be the “number of new cases reported today”, and negativeIncrease to be the number of new PCR tests minus positiveIncrease. But this is invalid because the former includes PCR and serology tests, while the latter includes only PCR tests. The correct value to use for negativeIncrease would be “Number of new tests reported today” which is higher since it includes serology tests. Then at least positiveIncrease and negativeIncrease would be reporting from the same collection of tests (i.e. both PCR and serology). Coincidentally, the fraction positive by PCR and by serology in Arizona happens to be similar at this time, due to the past and current trajectory of the epidemic. Hopefully in the future the Arizona website will offer a daily breakdown of PCR+ and PCR- tests, but at this time they only offer the daily total of PCR tests (and of serology tests), and then the weekly breakdown on a different section of the website (see “Laboratory Testing” in the link below).

Link to data source Site: https://www.azdhs.gov/preparedness/epidemiology-disease-control/infectious-disease-epidemiology/covid-19/dashboards/index.php Screenshot example: https://covidtracking.com/screenshots/AZ/AZ-20200723-180514.png

HI @rgerkin

We do not use scrapers to capture the data. We use actual humans that visit the websites daily.

Typically our positive number in Arizona includes both probable and confirmed cases, but we also separately publish in our API the number of PCR lab confirmed cases. This is slightly complicated by the fact that as of 7/18 Arizona no longer provides the breakdown of probable vs. confirmed cases via a hover-over action on their dashboard.

We do not include antibody tests as part of the totals. We capture them separately, and plan to publish them in the future.

muamichali commented on July 26, 2020 at 12:12 pm

Hi again @rgerkin

Thank you for your kind words.

API consumers can use PCR-only fields in our API (positiveCasesViral, totalTestsViral) when available from the states to calculate percent positive rate.

Since the number of probable cases in AZ is very small compared to the overall number of cases (as far as we can tell from when they still reported the number until 7/18) it does not result in skewing the PPR significantly.

We are aware of the fact that probable cases are included in the overall cases as I mentioned above & we are working to remove them (https://github.com/COVID19Tracking/issues/issues/578). Working with 56 different data sets from the different jurisdictions and within a mostly volunteer organization means that changes are not always immediate.

Regarding the suggestion to lump serology tests and cases for calculating the percent positive rate, we published a blog post about this topic recently: https://covidtracking.com/blog/antibody-data-reporting

@rgerkin: We also have a scraper that fetches data from states that enables an extra level of checks in addition to the human checkers. If you would like to contribute your PR to that effort, we would really appreciate it. https://github.com/COVID19Tracking/covid19-datafetcher
